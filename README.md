# CS250 8-2 Journal
Interpreting user requirements and implementing them in a program starts with a clear understanding of what the user wants to accomplish and why. In the SNHU Travel project, this was achieved through user stories. These stories translated customer requirements into executable development tasks. For example, the user story “As a customer, I want to search for travel packages so that I can compare options” conveys both the capability and the purpose. This made it easier for the development team to design, build, and test features that met user expectations. User stories also supported the Product Owner's prioritization, ensuring high-value features, such as booking and customer management, were developed first. By splitting requirements into smaller, testable increments with defined acceptance criteria, the team continuously delivered working software and stayed aligned with user needs.

When developing programs, I have learned the value of iterative, joint processes enabled by Agile methods. The Scrum framework in this project emphasized continuous improvement, adaptability, and communication. Practices such as sprint planning, daily stand-ups, sprint reviews, and retrospectives kept the team aligned. These practices also helped us respond quickly to changes, such as adding a promotional discount feature. Moving forward, I plan to use these Agile methods, including backlog prioritization, iterative development, and regular team communication. Being a good team member means contributing actively, communicating clearly, and supporting the team’s goals. It also means being adaptable, open to feedback, and willing to collaborate across roles. For instance, developers and testers worked together to resolve issues before the sprint review. Strong teamwork, accountability, and a commitment to improvement are essential for delivering high-quality software.
